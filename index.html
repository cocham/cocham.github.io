소셜 로그인 & 인증
로그인 흐름
1
소셜 로그인 요청
카카오·네이버·구글 중 선택 → 해당 서비스 인증 페이지로 이동
2
사용자 확인 & 저장
소셜 계정 인증 완료 → 프로필 정보 수신 → 신규 회원이면 DB 저장, 기존 회원이면 조회
3
토큰 발급
Access Token(단기) + Refresh Token(장기) 두 종류 발급. RT는 서버 Redis에, AT는 클라이언트 메모리에 보관.
4
Refresh Token 쿠키 저장
RT는 JS로 읽을 수 없는 쿠키에 저장해 탈취 위험을 줄임. 재발급 요청 시 브라우저가 자동으로 전송.
5
Access Token 재발급
AT가 만료되면 RT로 새 AT 발급. 이때 RT도 함께 교체해 탈취된 RT가 재사용되지 않도록 차단.
보안 처리
토큰 교체 (RTR)
AT 재발급 시마다 RT도 새로 교체. 누군가 RT를 훔쳐도 이미 무효화된 상태라 피해를 막을 수 있음.
로그아웃 처리
로그아웃 시 기존 AT를 블랙리스트에 등록. 이후 해당 토큰으로 요청이 들어오면 즉시 차단.
이상 징후 감지
서버가 저장한 RT와 요청된 RT가 다르면 탈취로 판단, 즉시 모든 토큰 삭제 후 재로그인 유도.
쿠키 보안
RT 쿠키는 JS 접근 차단 + HTTPS 전용 + 동일 사이트 요청만 허용으로 설정.

플레이리스트 등록 흐름
Spotify 링크 → PLYY 플레이리스트 생성
1
요청 수신
Spotify 링크, 큐레이션 코멘트, 태그, 커버 이미지를 함께 전송
2
Spotify에서 트랙 정보 수집
서버 토큰으로 Spotify API 호출 → 트랙 목록·썸네일·앨범 정보 파싱
3
중복 트랙 처리
이미 DB에 있는 트랙은 그대로 재사용, 새 트랙만 한 번에 저장. N+1 문제 방지.
4
플레이리스트 조립
트랙 순서·큐레이션 코멘트·장르·무드·태그를 묶어 하나의 플레이리스트 객체로 생성
5
저장 & 응답
전체 과정을 하나의 트랜잭션으로 묶어 중간 실패 시 전체 롤백. 완료 후 생성된 플레이리스트 ID 반환.
태그 자동 분류

입력된 태그를 장르 → 무드 → 커스텀 순서로 자동 분류합니다.

"힙합" → 장르 목록에 있나요? Yes → 장르
"새벽감성" → 무드 목록에 있나요? Yes → 무드
"털ㄴ업" → 어디에도 없음 → 커스텀
